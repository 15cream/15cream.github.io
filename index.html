[阅读|论文]BCFinder: A Lightweight and Platform-Independent Tool to Find Third-Party Components in Binaries

BibTeX:

阅读笔记

摘要

背景和问题

  1. 开源盛行；
  2. CBD(component-based development)和代码重用频繁；
  3. 以上背景引入了一些问题，例如license violation和security weaknesses；
  4. （原文）While code reuse detection has been extensively studied in source form, third-party components detection for software in binary form especially based on large scale database like Github has been less researched.

解决方案

  1. 采集了22K C/C++ repos on Github；
  2. Extend the code reuse detection for binaries against such a large-scale data set；
  3. 设计了一个系统BCFinder，可以通过特征匹配(feature matching)自动识别二进制文件中的第三方库组建。

评估

  1. 在大量真实的程序二进制文件上使用BCFinder，这些binaries是跨平台的，且编译选项不同（？）；
  2. 实验结果显示BCFinder是一个有效的二进制分析辅助工具；

总结

（原文）BCFinder is, by far, the first lightweight, rapid and platform-independent tool to detect component reuse in binaries against a large-scale data base like Github.

重点内容（个人偏向）

challenges

首先作者选择了feature matching作为third-party components detection的方法，且选择的feature具体为string constants。此时会遇到两个challenges。

Partially built binaries.
  1. 第三方组件中只有部分代码被包含到待测二进制中；
  2. 导致低匹配占比（a low matching proportion），进而导致false negatives；
  3. 对问题的定性：（原文）This is caused by an innate problem, for lacking of features.
  4. 解决方案：（原文）To identify incomplete components, we use reserved unique strings (strings appear only one time in all repos) in binaries to determine the existence of a component.
String duplication.
  1. 从binaries中提取的strings可能重复出现在不同的repos中，导致false positives；
  2. 解决方案：（原文）We have taken a series of measures to deduplicate from building the database to feature matching.

Strings Duplication and Deduplication

作者将Strings Duplication分为两类：

  1. Internal code clones in components. 即整个模块的包含，例如有三个模块都包含了第四个模块，导致第四个模块的unique strings出现在前三个模块中。作者会尝试识别出这样的情况。
  2. Cross-repetition between components. 简单的component之间存在重复的strings？作者会忽略在不同components之间出现次数超过16次的strings。

Matching method

  1. 对unique strings和duplicated strings区分对待。当binary与一个component的matched unique strings 和 matched duplicated strings 占比分别达到某个threshold时（会有两个thresholds），认为该component是有效的。
  2. （原文）After matching repos list is generated, we apply a post-process which deletes repos with low matching rate. From the last one in the matching list, we check one by one from back to front. If the majority of matched strings of a repo can be included in repos with higher matching rate, we think this repo is false positive as a result of strings duplication. Finally, we only show matching results with high matching ratio and high absolute matching quantity.

总结

  1. 可以了解一些Third-party component detection的背景知识，例如它在software security, commercial profit, software maintenance, and copyright protection这些方面的作用；
  2. 可以了解下作者对组件间相似问题的定义和解决方案。

定位通过Helper objects实现的callbacks

背景知识

我们首先看一下Helper objects的概念。

In object-oriented programming, a helper class is used to assist in providing some functionality, which isn’t the main goal of the application or class in which it is used[*citation needed*]. An instance of a helper class is called a helper object (for example, in the delegation pattern).

– the wikipedia of Helper class

In delegation, an object handles a request by delegating to a second object (the delegate). The delegate is a helper object, but with the original context. With language-level support for delegation, this is done implicitly by having self in the delegate refer to the original (sending) object, not the delegate (receiving object). In the delegate pattern, this is instead accomplished by explicitly passing the original object to the delegate, as an argument to a method.[1]

– the wikipedia of Delegation pattern

根据前面的描述，helper objects是实现callback的方法之一，即我们能够借助helper objects来实现对事件的响应。为了有更直观的感受，我们在Apple Developer Documentation中搜索delegate获得一些实际的案例。

UIApplicationDelegate

Every app must have an app delegate object to respond to app-related messages. For example, the app notifies its delegate when the app finishes launching and when its foreground or background execution status changes. Similarly, app-related messages coming from the system are often routed to the app delegate for handling. Xcode provides an initial app delegate for every app and you should not need to change this delegate later.

The delegate must adopt the UIApplicationDelegate formal protocol.

– the delegate(instance property) of the app object, Apple Developer Documentation 🔗

具体来说，开发者需要实现一个遵循UIApplicationDelegate协议的delegate来处理应用生命周期相关的事件。

UIScrollViewDelegate

The methods declared by the UIScrollViewDelegate protocol allow the adopting delegate to respond to messages from the UIScrollView class and thus respond to, and in some affect, operations such as scrolling, zooming, deceleration of scrolled content, and scrolling animations.

– the Apple Developer Documentation of UIScrollViewDelegate protocol 🔗

UITableViewDelegate

Use the methods of this protocol to manage the following features:

  • Create and manage custom header and footer views.
  • Specify custom heights for rows, headers, and footers.
  • Provide height estimates for better scrolling support.
  • Indent row content.
  • Respond to row selections.
  • Respond to swipes and other actions in table rows.
  • Support editing the table’s content.

– the Apple Developer Documentation of UITableViewDelegate protocol 🔗

默认情况下，所有的UITableViewController都会遵循该协议。开发者可以实现一个UITableViewController，并自行定义相关事件发生时的响应操作，例如在“- tableView:willSelectRowAtIndexPath:”方法中实现当表格的某一行被选中时应做如何处理。

确定目标

在iOS应用中，delegation是通过protocol实现的。而protocol的定义者通常可以分为两类，一类是Cocoa Frameworks，例如我们前面给的3个例子；另一类则是开发者，例如blog中给的例子。从逆向的角度来看，它们的差别主要有两个：

  1. Cocoa Frameworks定义的protocol通常有资料可查，因为它本来提供给广大开发者使用的。我们明确地知道，每一个protocol method是用来处理什么事件的。而开发者自行定义的protocol，通常并不是“流传甚广”的信息，即使是第三方库中开放的protocol，也不如前者来得广泛。
  2. callback的调用者位置不一样。对前者，callback的调用者通常不在app的可执行文件中，而在运行时链接的动态链接库中；对后者，callback的调用者通常在app的可执行文件中，即“调用delegate的method”这一行代码是开发者写在了应用中的。因此，当我们在构造应用的call graph时，第一类callback通常是作为图的entry point，而后者则是普通的、具有caller的节点。某种意义上，也可以理解为它们处理的事件类型不一样，前者处理系统事件，后者处理自定义事件。

Locate callbacks in iOS apps (Intro)

背景

什么是callbacks

In computer programming, a callback, also known as a “call-after“[1] function, is any executable code that is passed as an argument to other code; that other code is expected to call back (execute) the argument at a given time. This execution may be immediate as in a synchronous callback, or it might happen at a later time as in an asynchronous callback. Programming languages support callbacks in different ways, often implementing them with subroutines, lambda expressions, blocks, or function pointers.

–wikipedia

以上是callback的广泛概念，以及常见的实现方式。当落实到iOS应用中，根据该blog的总结，callbacks是用来响应事件的，且有4种具体的实现途径：

  1. Target-action/目标-动作对；
  2. Helper objects/辅助对象；
  3. Notifications/通告；
  4. Blocks。

为什么要定位callbacks

作为应用逻辑的突破口，便于开展分析。

iOS应用在使用过程中，大多数时间是事件驱动的。从callback开始分析：

  1. 可以确定应用是否实现了某些功能（或者说，是否会响应某些事件）；
  2. 更容易得到完整的事件响应链，便于梳理逻辑；
  3. 从二进制分析的角度，更容易得到完善的call graph。
